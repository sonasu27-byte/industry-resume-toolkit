## /改简历 — 主命令:按方法论优化你的简历,直接出成品

对 `$ARGUMENTS` 这份简历(或用户拖进聊天框的简历)执行完整的优化流程。

### 这个命令的定位

你(用户)上传简历,**AI 直接按方法论改完输出成品**。不分服务档位,不区分"给方向 vs 全程改"——AI 看你简历的信息密度,**自己判断**要不要追问你挖经历,然后直接出最终版本。

### 流程

#### Step 1 — 准备

1. **读 skill 主文件 + patterns**(都在 `~/.claude/skills/industry-resume-toolkit/`):
   - `SKILL.md`(主要看 §2 三层框架、§3 简历类型分类、§5 改简历的执行流程、§6 保真原则)
   - `patterns/通用.md`(必读)
   - 根据用户的 `profile.json` 选企业性质和岗位:
     - `patterns/企业性质/<大厂|国企|银行|外企>.md`
     - `patterns/岗位/<对应岗位>.md`(如有)
   - 对应经验密度的示例:`examples/经验充足/案例分析.md` 或 `examples/经验较少/案例分析.md`

2. **检查 profile**:
   - `~/.claude/skills/industry-resume-toolkit/profile.json` 存在 → 读它
   - 不存在 → 提示用户先跑 `/诊断`,或者**在本命令里快速补问 6 个问题**(就别让用户两次输入了)

3. **归位输入文件**(保证 inputs/ 和 outputs/ 数据完整):
   - 简历来自 `$ARGUMENTS` 路径或用户拖进聊天框的附件
   - 如果还不在 `~/.claude/skills/industry-resume-toolkit/inputs/`,复制/保存一份过去
   - 用 bash `cp` 处理。**二进制 docx/pdf 必须保留原始字节**
   - 如果已经在 inputs/,跳过

#### Step 2 — 读 + 评估

1. **读用户的简历**(从 inputs/ 那份)

2. **判断经验密度**:
   - **经验充足型**:2 段以上正式实习,或有大厂经历
   - **经验较少型**:实习 ≤1 段,或都是小公司/校园活动
   - 是否叠加转行(基于 profile)

3. **评估信息密度**(决定是否需要 Phase 3 挖经历):
   - 出现 3+ 处模糊量化("很多 / 大量 / 较多 / 诸多 / 多个 / 丰富")
   - 关键实习段缺少业务背景描述
   - 项目经历没有量化结果
   - 重要经历只有 1 行职责
   - 满足以上任一条件 → 触发 Phase 3 挖经历

#### Step 3 — 决策分支

**分支 A:信息充分,直接改**

跳到 Step 4。

**分支 B:信息缺失,先挖经历**

输出"我需要先问你几个问题再改"+ 一次只问 **3-5 个最关键的问题**(不要塞 30 个让用户崩溃),按这个结构组织:

```markdown
# 改简历前,我需要先问你几个关键问题

为了不在简历里编造数字 / 模糊描述,我需要你回答这几个问题再开始改。

## 关于 [实习段 1 / 公司名]

1. [具体问题 1] — 比如"你大约处理了多少笔抽凭工作?估算就行,几百到几千都可以"
2. [具体问题 2]
3. [具体问题 3]

## 关于 [实习段 2 或最关键的项目]

4. [具体问题 4]
5. [具体问题 5]

回答完我就开始改。**不知道的可以说"不知道"**,我会用"建议补充"的方式标出来,不会编。
```

**暂停**,等用户答复。用户答完进入 Step 4。

#### Step 4 — 改简历

按三层框架执行(在原 .docx 上改,或从 PDF/纯文本转 Word):

1. **Phase 1 信息卫生**:按 §2 Phase 1 清单逐条排查 + 修正
2. **Phase 2 业务价值翻译**:按 §2 Phase 2 + 对应 patterns 重写
3. **Phase 3 经历挖掘**(如适用):把用户答复整合进简历

**保真原则(铁律)**:用户没确认的数字坚决不写。title 可调,职责描述不能编。

**输入格式处理**:
- **`.docx`** → 在原文件基础上改,保留模板/字体/排版(优先方案)
- **`.pdf`** → 用 Python 提取文字,转成 .docx 输出。**不要让用户提供 Word**,直接转
- **`.md` / 纯文本** → 转成 .docx 输出
- 如果用户在 inputs/ 里放了自己想要的模板文件(不是简历本身),按那个模板格式

**.docx 在原文件上改的实现**:
```python
from docx import Document
import shutil
import os

skill_root = os.path.expanduser('~/.claude/skills/industry-resume-toolkit')
input_file = f'{skill_root}/inputs/原文件.docx'
output_file = f'{skill_root}/outputs/<原文件名>-改后.docx'

shutil.copy(input_file, output_file)
doc = Document(output_file)

# 遍历段落 + 表格,保留 run 格式,只改文字
for para in doc.paragraphs:
    if para.text.strip() == "原文":
        if para.runs:
            first_run = para.runs[0]
            for run in para.runs[1:]:
                run.text = ''
            first_run.text = "改后的文字"

for table in doc.tables:
    for row in table.rows:
        for cell in row.cells:
            for para in cell.paragraphs:
                # 同上
                pass

doc.save(output_file)
```

```python
# ... 按解析结果写入,使用通用专业排版(Calibri/微软雅黑/合理字号)

```

#### Step 5 — 输出 + 询问格式

1. **默认成品**:`~/.claude/skills/industry-resume-toolkit/outputs/<原文件名>-改后.docx`

2. **在对话里简短汇报**:

```markdown
# 简历优化完成 ✓

**你的画像**:经验充足型 / 较少型(+ 转行?) | 目标:[岗位] @ [企业性质] | 中/英文

**这次主要改动**(按重要性):

1. **[改动 1 的核心]** — 简短说明
2. **[改动 2]**
3. **[改动 3]**
4. **[改动 4]**
5. **[改动 5]**

**走了哪几层**:
- Phase 1 信息卫生:[轻 / 中 / 重]
- Phase 2 业务价值翻译:[轻 / 中 / 重]
- Phase 3 经历挖掘:[做了 / 没做 — 信息够]

**成品**:`outputs/<原文件名>-改后.docx`

---

要不要换其他格式?我可以给你:
- 📄 Markdown 版(方便复制粘贴 / 在线投递)
- 📑 PDF 版(直接投递)
- 🌐 英文版 / 双语对照版
- 🎨 用你自己的模板重排(如果你有更喜欢的简历模板,可以提供文件)

或者你想改哪里继续告诉我。
```

3. **如果用户要其他格式**,生成对应的版本(Markdown / PDF / 翻译版等)。

### 重要原则

- **用户没确认的数据坚决不写** — 严格遵守 SKILL.md §6 保真原则
- **PDF 输入直接转,不要要求 Word** — 这是用户最讨厌的细节
- **不要自带模板** — 默认用 `.docx` 通用专业排版,用户想要特定模板让他自己提供
- **第二人称称呼用户** — "你的简历"、"你的目标",不要用"客户"
- **挖经历问题精炼** — 一次 3-5 个,不要塞 30 个把用户问崩溃
- **改完直接出成品** — 不要让用户跑两个命令(挖经历 + 改)。用户的预期是 `/改简历` 跑一遍出结果

### 兜底场景

- 用户没跑 `/诊断`:本命令里快速问 6 个问题,然后继续
- 用户 profile 跟简历不匹配(eg profile 写目标"大厂",但简历像在投国企):温和提醒,确认意图
- 用户简历严重不完整(连基础信息都没有):提示"这种情况更适合 0-1 从头写,本工具优化已有简历"
